# Property Graph Schema Design for Regulatory Compliance (ARIA)

## Concept definition

A **property graph schema** is the set of conventions that govern *what* may exist in the graph: node labels, relationship types, property keys, cardinality expectations, and identity rules. Unlike a rigid SQL DDL that often fixes columns per table, a graph schema is frequently **semantic and contractual**: enforced by application code, constraints, and review rather than by a single `CREATE TABLE` for every fact.

For regulatory compliance, a useful schema balances:

- **Fidelity to legal structure** (regulation → article → requirement).
- **Operational reality** (systems, teams, policies, deadlines).
- **Query ergonomics** for multi-hop questions and GraphRAG expansion.
- **Idempotent ingestion** so the same source document can be merged without duplicate entities.

ARIA expresses this schema as Python enums and data classes (`NodeLabel`, `EdgeType`, `GraphNode`, `GraphEdge`) and as Neo4j DDL (uniqueness constraints and indexes) generated from a single source of truth.

## Why it matters

Poor graph modeling leads to **query explosions**, **ambiguous merges**, and **LLM context that looks structured but is wrong**. In compliance settings, errors propagate into risk assessments and remediation tasks.

Good schema design:

- Makes **ownership and coverage** explicit (who owns a system, whether a requirement is addressed by a policy).
- Separates **legal text structure** from **enterprise objects** (internal systems are not articles).
- Provides **stable merge keys** so agents and ingestion pipelines can upsert the same logical entity across runs.
- Restricts relationship types to a **closed set** of valid `(source, relationship, target)` patterns, reducing garbage edges from extraction noise.

ARIA’s schema is intentionally small and navigable for a portfolio demonstration while remaining representative of real compliance graphs.

## How it is implemented in this repo

### Single source of truth: `aria/contracts/graph_entities.py`

`SCHEMA_VERSION` documents the contract revision (`"0.1.0"` at time of writing).

**Node labels** (`NodeLabel` enum):

| Enum member | Neo4j label | Typical role |
|-------------|-------------|--------------|
| `REGULATION` | `Regulation` | Legal instrument root |
| `ARTICLE` | `Article` | Contained section imposing or describing rules |
| `REQUIREMENT` | `Requirement` | Normalized obligation |
| `POLICY_DOCUMENT` | `PolicyDocument` | Internal control or policy text |
| `INTERNAL_SYSTEM` | `InternalSystem` | In-scope application or platform |
| `TEAM` | `Team` | Organizational owner |
| `JURISDICTION` | `Jurisdiction` | Applicable region or regime |
| `DEADLINE` | `Deadline` | Time-bound milestone linked to articles |

**Edge types** (`EdgeType` enum) define the directed vocabulary:

- `CONTAINS` — Regulation → Article (hierarchical containment).
- `IMPOSES` — Article → Requirement (legal text to obligation).
- `AMENDS`, `REFERENCES` — Regulation → Regulation (cross-instrument graph).
- `APPLIES_IN` — Regulation → Jurisdiction.
- `AFFECTS` — Requirement → InternalSystem (impact).
- `ADDRESSED_BY` — Requirement → PolicyDocument (coverage).
- `OWNED_BY` — PolicyDocument → Team and InternalSystem → Team.
- `HAS_DEADLINE` — Article → Deadline.

`GraphWritePayload` batches `nodes` and `edges` for atomic writes; `GraphWriteStatus` reports created/merged counts and errors. `GraphQueryResult` types MCP and API responses.

### Constraints, merge keys, and valid edges: `aria/graph/schema.py`

`NODE_MERGE_KEYS` maps every `NodeLabel` to `"id"`. That design choice means:

- **One canonical business key** per entity type, simplifying agent instructions and extraction prompts.
- **Uniqueness constraints** generated by `generate_constraint_statements()` — for each label, `n.id IS UNIQUE`, preventing duplicate regulations or requirements with the same logical id.
- **MERGE operations** in the graph writer (elsewhere in the codebase) can use `id` as the deterministic upsert key, supporting **idempotent** re-ingestion of the same regulation.

`VALID_EDGES` is a list of allowed triples `(source_label, EdgeType, target_label)`. This is a lightweight **schema guard**: only these patterns should be produced by extractors or tools. It documents the intended **ontology** for readers and can be validated in tests or writers.

`generate_index_statements()` adds secondary indexes on high-selectivity lookup fields:

- Regulation `title`, Article `number`, Requirement `obligation_type`, InternalSystem `category`, Team `name`.

These align with filter-and-sort patterns in UI and agent tools without replacing the primary `id` merge discipline.

### Modeling decisions in depth

**Why separate `Deadline` nodes instead of only date properties on `Article`?**

Deadlines can be shared, revised, or referenced in explanations as first-class events. Modeling them as nodes:

- Allows a consistent `NodeLabel` pattern and the same merge-key machinery as other entities.
- Supports `HAS_DEADLINE` as an explicit relationship that GraphRAG expansion can traverse.
- Avoids overloading article properties with multiple dates (entry into force, reporting date, transitional periods) when those need separate identity or provenance later.

If deadlines were always a single scalar per article with no shared or evolving semantics, a property might suffice; ARIA favors **navigable obligations** for demonstration clarity.

**Why MERGE keys and uniqueness constraints?**

Ingestion and agents may run repeatedly against overlapping corpora. Without merge keys, every run creates duplicates; vector metadata and graph nodes then **diverge**. `id` plus Neo4j uniqueness constraints ensure one node per logical entity and stable anchor ids for `HybridRetriever` (see `aria/retrieval/hybrid_retriever.py`, which reads `node_id` from chunk metadata).

**Why separate `InternalSystem` and `Team`?**

Systems and org units are different **real-world kinds** with different attributes (category, environment, vs. team name, cost center). Requirements **affect** systems; systems and policies are **owned by** teams. Collapsing into a generic `Entity` node would reduce typing safety in Cypher (`MATCH (n)` everywhere) and obscure business questions such as “uncovered requirements per team via system ownership.” The schema keeps traversals **typed and readable**.

**Why `PolicyDocument` between `Requirement` and coverage?**

“Addressed by” is a document-level assertion, not a team-level one. Separating policy from team allows queries that find **gaps** (`NOT (req)-[:ADDRESSED_BY]->()`) independent of ownership, and separate queries for **who must fix** the gap via `OWNED_BY`.

### Ingestion coupling

`aria/ingestion/pipeline.py` accepts a `graph_writer` that consumes structured `ExtractedEntities` (see `aria/contracts/regulation.py` for extraction shape). Successful writes set `graph_written` on `IngestionResult`. The same document produces **chunks** for Chroma and **entities** for Neo4j, so metadata can carry `node_id` consistent with graph `id` values.

### Authoritative `VALID_EDGES` listing

The following triples are the **only** directed patterns enumerated in `aria/graph/schema.py` as `VALID_EDGES`. They should be treated as the canonical ontology for writers and extractors:

| Source label | Relationship | Target label |
|--------------|--------------|--------------|
| `Regulation` | `CONTAINS` | `Article` |
| `Article` | `IMPOSES` | `Requirement` |
| `Regulation` | `AMENDS` | `Regulation` |
| `Regulation` | `REFERENCES` | `Regulation` |
| `Regulation` | `APPLIES_IN` | `Jurisdiction` |
| `Requirement` | `AFFECTS` | `InternalSystem` |
| `Requirement` | `ADDRESSED_BY` | `PolicyDocument` |
| `PolicyDocument` | `OWNED_BY` | `Team` |
| `InternalSystem` | `OWNED_BY` | `Team` |
| `Article` | `HAS_DEADLINE` | `Deadline` |

Any new domain relationship (for example `Requirement`–`Requirement` dependencies) should be added here **before** extractors emit edges, so constraints, tests, and documentation stay aligned.

### Indexes and query ergonomics

Secondary indexes generated in `generate_index_statements()` support **human-facing** lookups (title search, sorting articles by number, filtering systems by category) while keeping **`id`** as the technical merge key. This split mirrors common compliance UIs: analysts search by name or article number, but APIs and agents refer to stable identifiers.

### Contract typing for agents and MCP

`GraphQueryResult` and `GraphWritePayload` provide **typed boundaries** between Neo4j drivers, orchestration, and tool surfaces. When MCP tools return graph rows, consumers can rely on column names and row dictionaries without parsing ad hoc JSON shapes. That consistency matters for multi-agent workflows where one agent writes and another reads the same schema.

## Tradeoffs

**Explicit labels vs generic “Thing” nodes**

- Labels improve query clarity and index use; too many labels fragment the model. ARIA uses a compact set aligned to compliance language.

**Strict `VALID_EDGES` vs fully open graph**

- Closed sets catch extraction mistakes early but require code updates when the domain grows (e.g., `SUPERSEDES` edges).

**Single merge key `id` vs composite keys**

- Simplicity wins for demos; production systems might use `(jurisdiction, instrument_number)` composites — Neo4j supports constraints on multiple properties if needed.

**Deadline as node vs property**

- Node-per-deadline adds traversal hops; property-only is smaller. ARIA chooses expressiveness for GraphRAG and narrative traceability.

**Versioning regulations**

- The schema supports `AMENDS`/`REFERENCES` between `Regulation` nodes but does not, by itself, encode temporal validity on every edge. Production systems often add `effective_from` / `effective_to` properties or version nodes; ARIA keeps the demo schema minimal.
